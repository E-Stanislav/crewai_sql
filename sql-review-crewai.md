# SQL Review Crew (на базе CrewAI)

## Цель проекта

Создать оффлайн-мультиагентную систему на базе CrewAI, которая автоматически проходит по каталогу с большим количеством SQL-файлов (2000+), проверяет каждый запрос на валидность, соответствие схеме БД, производительность и читаемость, а также при необходимости предлагает переписанную, более оптимальную версию запроса.

Система должна работать как линтер/ревьюер для аналитических и продуктовых SQL-запросов, дополняя классический оптимизатор СУБД, а не заменяя его.

## Выходной результат (артефакты)

Для каждого обработанного SQL-файла ожидается структурированный результат, который может включать:

- Краткий статус: `OK`, `WARNING`, `ERROR`.
- Список найденных проблем и рекомендаций:
  - синтаксические и семантические ошибки;
  - несоответствия схеме (отсутствующие таблицы/колонки, неверные типы);
  - потенциальные проблемы производительности (full scan, неиспользуемые индексы, избыточные JOIN/CTE и т.п.);
  - замечания по стилю и читаемости.
- Предложенная улучшенная версия запроса (если применимо) в блоке `sql`.
- Мета‑данные анализа: используемая БД, длительность выполнения EXPLAIN/ANALYZE, оценочная стоимость плана, хэш исходного запросa.

Агрегированно по каталогу система должна уметь формировать отчет (Markdown/HTML/JSON) с:

- рейтингом самых «тяжелых» запросов;
- сводкой по типам найденных проблем;
- рекомендациями по индексам, материализованным представлениям и паттернам переписывания.

## Базовая архитектура на CrewAI

Проект строится вокруг CrewAI как фреймворка мультиагентной оркестрации.

Основные компоненты:

- **Agents (агенты)**
  - *File Scanner Agent* — читает структуру каталогов, собирает пути к `.sql` и создаёт задачи на анализ.
  - *Schema & Syntax Agent* — валидирует запрос относительно схемы БД (через кастомный DB‑tool или готовые database‑tools CrewAI) и проверяет базовый синтаксис.
  - *Plan & Optimization Agent* — вызывает `EXPLAIN/EXPLAIN ANALYZE` в целевой СУБД, анализирует план и, используя LLM, предлагает оптимизации.
  - *Style & Best‑Practices Agent* — оценивает читаемость, единообразие стиля, наличие антипаттернов (SELECT *, громоздкие подзапросы и т.д.).
  - *Refactor Agent* — на основе выводов других агентов генерирует улучшенную версию SQL‑запроса с сохранением семантики.
  - *Report Aggregator Agent* — собирает результаты, агрегирует их и формирует итоговый отчёт по файлам и всему каталогу.

- **Tasks (задачи)**
  - Задача анализа одного SQL‑файла: вход — содержимое файла и контекст БД; выход — структурированный JSON/Markdown‑отчёт + переписанный запрос.
  - Задача генерации общего отчета по каталогу.

- **Tools (инструменты)**
  - Файловые инструменты CrewAI (`DirectoryReadTool`, `FileReadTool`) для обхода каталогов и чтения SQL‑файлов.[web:16][web:22]
  - Database‑tools CrewAI (например, `MySQLTool`, `PGSearchTool`, `NL2SQLTool`) либо кастомные инструменты для подключения к Postgres/ClickHouse/Vertica и выполнения EXPLAIN/ANALYZE.[web:15][web:18][web:24]
  - Собственные кастомные инструменты (через `@tool`/`BaseTool`) для:
    - вызова EXPLAIN/ANALYZE;
    - получения схемы и статистики (`information_schema`, `pg_catalog`, системные таблицы);
    - кеширования результатов анализа по хэшу запроса.

- **Process (процесс)**
  - Основной процесс — последовательный или иерархический pipeline: `Schema & Syntax → Plan & Optimization → Style → Refactor → Report Aggregator`, запуск задач по файлам может быть параллельным.

## Технологический стек

- Python 3.11+
- CrewAI (agents, tasks, processes, tools).[web:3][web:19][web:23][web:25]
- crewai-tools (файловые и database‑tools).[web:16][web:18][web:27]
- Драйверы СУБД (psycopg2/asyncpg, clickhouse‑driver и т.п.).
- LLM‑провайдер (OpenAI/Anthropic/локальная модель).

## Формат минимального MVP

MVP‑версия должна уметь:

1. Пройти по указанной папке и собрать все `.sql` файлы.
2. Для каждого файла запустить crew с агентами `Schema & Syntax`, `Plan & Optimization`, `Style` и `Report Aggregator`.
3. Сохранить для каждого файла Markdown‑отчёт `*.sql.review.md` рядом с исходником или в отдельной папке `reports/`.
4. Сформировать общий `SUMMARY.md` с агрегированной статистикой и ссылками на отчёты по отдельным файлам.
